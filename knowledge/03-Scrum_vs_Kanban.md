# Fiche 03: Scrum vs Kanban

## 1. Tableau Comparatif

| Critère | Scrum | Kanban |
| :--- | :--- | :--- |
| **Rythme** | **Sprints** fixes (ex: 2 semaines). | **Flux continu** (Flow). Pas de sprint. |
| **Rôles** | Définis (Product Owner, Scrum Master, Dev Team). | Pas de rôles imposés, tout le monde participe au flux. |
| **Changement** | On ne change pas le scope *pendant* le sprint. | On peut changer les priorités à tout moment si la capacité le permet. |
| **Métriques** | Vélocité (Points de story). | Lead Time & Cycle Time (Temps de traversée). |
| **Usage DevOps** | Bien pour les projets de développement (features). | **Idéal pour l'Ops/Maintenance** (tickets, bugs, support). |

## 2. Zoom sur Scrum
*   **Cérémonies** :
    *   *Daily Standup* : Synchro rapide (15min). "Qu'ai-je fait hier ? Que fais-je aujourd'hui ? Bloquants ?".
    *   *Sprint Planning* : On définit ce qu'on embarque dans le sprint.
    *   *Review* : Démo du produit fini.
    *   *Retrospective* : **Cruciale**. Comment améliorer notre façon de travailler ? (Amélioration continue).
*   **Artefacts** : Backlog (liste de tout), Sprint Backlog (liste du sprint), Incrément (livrable).

## 3. Zoom sur Kanban (Recommandé pour DevOps)
Kanban est souvent préféré en DevOps car les urgences de production ne peuvent pas attendre "le prochain sprint".
*   **Principes Clés** :
    1.  **Visualiser le travail** : Un tableau (To Do, Doing, Done).
    2.  **Limiter le WIP (Work In Progress)** : "Stop starting, start finishing". Ne pas avoir 10 tâches en cours en même temps.
    3.  **Gérer le flux** : Identifier les goulots d'étranglement (Bottlenecks).

## 4. Scrumban
Un hybride souvent utilisé : on garde les rituels de Scrum (Retro, Daily) mais on travaille en flux continu comme Kanban sans sprints rigides.
